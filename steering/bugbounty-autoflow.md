## Subagent return contract (MANDATORY â€” prevents lost work)

How results reach the parent: when a subagent finishes, the coordinator receives ONLY that subagent's final assistant text message (its summary). If a subagent run ends on a tool call, an error, a timeout, or by exhausting its turn budget mid-task, the coordinator receives `No result` and ALL of that subagent's work is discarded â€” even if it found something real. A `No result`/empty return is a delivery failure, never proof a surface is clean.

Every subagent (recon, every hunter, scope loaders, researchers, the triager, the account manager) MUST therefore:
- Checkpoint as you go, not at the end. Write findings, payloads, decisive evidence, and artifact paths to `reports/<program>/...` and refresh the relevant Memory MCP key incrementally. Never hold a decisive result only in conversation context.
- Wrap up before running out of turns. If you are many tool calls deep, blocked, or only partly done, STOP starting new tool rounds and write your summary now. A proactive partial summary always beats being cut off mid-loop.
- Always end on a plain-text summary, never on a tool call. Structure it as: what was tested, decisive evidence (with artifact paths), current status (`validated` / `candidate` / `needs-more-evidence` / `blocked`), and ranked next actions.
- Never return empty. If blocked or incomplete, still return the blocker, your assumptions, what you checkpointed (with paths), and the next exact step. Returning a blocker is success; returning nothing is a delivery failure.

Quality guardrails:
- Do not let triage block broad exploration. Triage only candidate findings to avoid reducing discovery rate.
- Keep scope loading reliable and complete: use `@h1-scope-loader` for scope extraction, not high-cost deep analysis agents.
- Do not force account creation unless testing genuinely requires authenticated access.
- `@recon` is a mapper, not the main exploit worker. Once a lead clearly belongs to a specialist class, `@recon` must capture the decisive evidence, update structured recon artifacts/memory, and hand off within one short confirming batch instead of continuing the exploit branch itself.
- A short confirming batch for `@recon` means one focused request family, one focused browser batch, or a small bounded command batch to verify the surface classification. If that batch is complete and the lead is still specialist-owned, recon must stop local exploit work and hand off.
- AI/chatbot/agent/MCP surfaces are not ordinary recon leads. Once identified, they should be routed to `@llm-hunter` immediately after the minimum confirming evidence is captured.
- `@recon` must not write a takeover candidate on weak evidence like one-off `000`, generic timeouts, or a single failed fetch. Write to `takeover_candidates.txt` only when you have a provider-specific orphaning signal or at least two independent confirming indicators. Put weaker signals into a separate holding file such as `takeover_weak_signals.txt`.
- Use `@payload-researcher` intentionally. It should improve hit rate for scenario-specific payload and wordlist selection and increase coverage quality, not delay baseline probes that are already fully covered by standard defaults.
- `@payload-researcher` must answer the exact scenario it was asked about. It must not recap the whole hunt, reprioritize unrelated branches, or end with a user-facing option menu. If broader hunt context matters, include it as one short `Relevant Context` note and still return the exact payload or wordlist deliverable requested.
- If `@payload-researcher` returns a recap, a broad hunt summary, or a menu of next-step choices instead of the exact payload, wordlist, commit, or tooling deliverable requested, treat that run as failed scope control. Re-issue the task once with a narrower exact objective and required output sections instead of summarizing the bad answer.
- When `@payload-researcher` returns payloads, wordlists, request bodies, or reusable commands longer than a few lines, it must save them into local artifact file(s) and return the exact artifact path(s). Parent agents should treat a summary-only answer with no artifact path as incomplete when a reusable payload deliverable was requested.
- For passive retrieval of public JS bundles, source maps, static assets, and public documentation, prefer browser-live MCP, GitHub MCP, `curl`, or direct raw-file fetches. Do not use OpenCode's built-in `webfetch` or `websearch`; they are disabled so web access stays routed through MCP/browser tooling. Reserve Burp for HTTP mutation, replay, differential testing, and evidence capture where raw interception is actually needed.
- If a tool materially improves coverage, validation quality, or speed and it is missing, install the latest stable release from an official source or trusted package manager and use it with scenario-appropriate flags after checking current docs or `--help`.
- SecLists is installed at `/usr/share/seclists`. Prefer focused maintained lists from there before downloading ad hoc copies of the same wordlists.
- Prefer current, well-maintained tooling when it materially improves findings, but avoid churn from installing tools that do not help the exact scenario.
- If a tool supports authenticated or API-key-backed modes, use only credentials that already exist, are target-owned, or can be legitimately created inside the authorized workflow. If first-party test credentials are needed, call `@test-account-manager`. Do not obtain unrelated third-party API keys.
- `@test-account-manager` uses browser-live MCP exclusively for ALL browser work â€” discovery, multi-step create/verify/login, password reset, invite acceptance, OTP, and inbox-confirmation flows. Open an isolated context per target with `new_page` + `isolatedContext` to keep sessions separate. Take one `take_snapshot` to get element UIDs and reuse them with `fill`/`fill_form`/`click`/`wait_for`/`evaluate_script` until the DOM changes; prefer `evaluate_script` for complex DOM extraction over repeated snapshots. Use `list_network_requests`/`get_network_request` to capture tokens, cookies, and API responses. Do NOT spawn standalone Chrome, Zendriver, Playwright, Puppeteer, or any other browser process, the browser-live Chrome is already running, use it. If a flow is blocked even through browser-live (hard anti-bot/CAPTCHA), sleep for 3 minutes over and over agin untill the operator solves it manually for you.
- In browser-live, verify that the selected page or tab still matches the intended host, URL family, and auth state before acting. If the wrong page is selected, re-select the right page or open a fresh isolated context instead of continuing in the stale one.
- When Gmail or inbox work yields a decisive OTP, magic link, invite URL, or verification code, write that extracted value once to a stable local artifact and reuse that artifact instead of rereading the same full HTML message unless the message materially changed.
- When `@llm-hunter` or another specialist needs multi-step account creation, login bootstrap, verification, OTP handling, or inbox polling, it may do only one short confirming setup batch itself. After that, it must hand the workflow to `@test-account-manager` and stay focused on the exploit logic.
- If scope data is ambiguous, pause active exploitation and ask for clarification instead of guessing scope.
- If a specialist is close to step cap, checkpoint current evidence, payloads tried, and next 3 actions before it stops.
- If cap is reached near a promising lead, immediately re-invoke the same specialist with the checkpoint and continue from there.
- If a subagent fails, errors, times out, or returns empty or no actionable output, restart it. Re-invoke the same subagent with the same objective, full context, and an explicit note that the prior run failed or returned empty. On a second empty or failed run, restart once more with adjusted inputs (narrower objective, different entry angle, refreshed artifact paths, or corrected tool arguments). Only after 3 consecutive failed or empty runs on the same objective, record the blocker and route to a different specialist or branch. Never treat an empty subagent result as proof there is nothing to find; an empty or missing return is a delivery failure, not a clean result.
- For heavy tool-driven specialists, work in bounded batches. After each batch of one protocol family, one exploit family, or roughly 10-15 crafted requests or payload families, checkpoint the evidence, the current conclusion, the next best branch, and the stop condition before continuing.
- Do not spend a single model turn on long speculative synthesis after a large batch of tool output. Emit a compact structured checkpoint first, then continue with the next bounded batch.
- Treat unexpected success in a negative-path test as a high-priority signal for deeper analysis.
- Use MCPs intentionally: do not spam browser automation when raw HTTP is enough, and do not skip browser checks for client-side logic issues.

- Store strong payload families, wordlist choices, and stack-specific fuzz inputs in Memory MCP when they are high-signal and reusable.
- Any subagent that reaches a concrete finding checkpoint must write or refresh it in Memory MCP immediately instead of waiting for task end.
  - A concrete finding checkpoint means a candidate or validated issue with an exact asset or flow, weakness class, auth state, proof signal or deterministic differential, impact hypothesis, and an evidence or report path that would be costly to lose to compaction, crashes, step caps, delegation, or branch switching.
  - Use one stable finding key per issue, such as `<program>|<asset>|<weakness>|<auth-state>|<surface>`, and update the existing entry instead of creating duplicates for the same finding.
  - Store only the decisive summary: status (`candidate`, `validated`, `needs-more-evidence`, `invalidated`), minimal proof, one control or disconfirmation result when available, impact summary, blockers, next exact validation step, and artifact or report paths.
  - Do not store full raw request or response dumps, huge payload blocks, or live secrets in memory. Store redacted identifiers and artifact paths instead, unless no safer artifact exists and the exact value is required to continue.
- Treat public GitHub or social-media results as lead generation only. Validate every lead directly on the authorized target before concluding anything.
- Keep report artifacts structured and local-only. Every retained finding gets its own folder with the required field files plus evidence in `files/`.
- Keep session size under control: avoid pasting full raw request/response bodies by default. Store focused evidence (endpoint, method, status, key headers, key reflection markers, and relevant excerpts), and include full raw data only for high-signal findings.
- Avoid repeating identical payload/result blocks; summarize repeated negatives in structured tables or counts.
- When a tool emits very large output, keep the decisive lines plus an artifact path, not the entire dump, unless the full body is directly needed to prove the finding.
- Keep outputs reproducible and evidence-first.
